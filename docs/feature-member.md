# 회원 기능 모듈 설계

> Vori 의 회원 기능 (가입·로그인·프로필·탈퇴) 단일 모듈. 다른 모듈(지출 입력, 합리성 판정 등)은 본 모듈에서 인증된 사용자 ID 만 받음.

## 한 줄 요약

**시연 20명 + 친구 100명까지 안전하게 돌릴 수 있는** 최소 인증 + 프로필. 게임화(스탯·레벨)는 회원 모델에 처음부터 포함.

---

## 주요 기능 (Core — 졸작 시연에 반드시)

| # | 기능 | 설명 |
|---|------|------|
| 1 | **회원가입** | 이메일+비밀번호 또는 OAuth (회의 결정) |
| 2 | **로그인 / 로그아웃** | 세션 발급·해제 |
| 3 | **세션 유지** | 새로고침해도 로그인 상태 유지 |
| 4 | **프로필 조회·수정** | 닉네임·아바타 변경 |
| 5 | **회원 탈퇴** | 본인 데이터 일괄 삭제 |
| 6 | **사용자별 데이터 격리** | 다른 사용자 지출·통계 절대 접근 불가 |

## 부가 기능 (Additional — 우선순위 ↓ )

| # | 기능 | 우선순위 | 비고 |
|---|------|---------|------|
| 1 | **비밀번호 재설정** (이메일 OTP) | 중 | 이메일+비밀번호 채택 시 필요 |
| 2 | **소셜 로그인** (Google, Kakao) | 중 | OAuth 라이브러리 사용 — NextAuth 등 |
| 3 | **데이터 내보내기/삭제** (개인정보 보호) | 중 | 졸작 어필 + 법적 안전 |
| 4 | **친구 초대 링크** | 하 | Phase 2 (졸작 후) — vori 의 소셜 확장 |
| 5 | **2단계 인증** (TOTP) | 하 | 졸작 범위 초과 가능성 |
| 6 | **알림 설정** (푸시·이메일) | 하 | 모바일 RN 단계에서 |

## 사용자 흐름 (5개 시나리오)

### 1) 첫 가입
```
[랜딩] → [회원가입 폼] → 입력·제출 → [이메일 인증]* → [닉네임 설정] → [메인]
                                  ↑ OAuth 선택 시 스킵
```

### 2) 재로그인
```
[랜딩] → [로그인 폼] → 자격 확인 → [메인]
                          ↓ 실패
                       [에러: 5회 시 잠금]
```

### 3) 로그아웃
```
[설정] → [로그아웃] → 세션 해제 → [랜딩]
```

### 4) 프로필 수정
```
[설정] → [프로필] → 닉네임·아바타 변경 → [저장] → 토스트 "변경됨"
```

### 5) 회원 탈퇴
```
[설정] → [회원 탈퇴] → 비밀번호 재확인 → 안내 문구 동의 → [모든 데이터 삭제] → [랜딩]
```

## 데이터 모델 (ER)

```
┌────────────────────────────┐
│ users                       │
├────────────────────────────┤
│ id              PK          │
│ email           UNIQUE      │
│ password_hash               │ ← bcrypt 해시 (OAuth 만이면 NULL)
│ nickname                    │
│ avatar_url      NULLABLE    │
│ level           DEFAULT 1   │ ← 게임화 (vori 핵심)
│ total_score     DEFAULT 0   │
│ created_at                  │
│ updated_at                  │
│ deleted_at      NULLABLE    │ ← Soft delete
└────────────┬───────────────┘
             │ 1
             │
             │ N
┌────────────┴───────────────┐    ┌────────────────────────────┐
│ sessions                    │    │ oauth_providers (옵션)      │
├────────────────────────────┤    ├────────────────────────────┤
│ id              PK          │    │ id              PK          │
│ user_id         FK          │    │ user_id         FK          │
│ token_hash                  │    │ provider        ('google',  │
│ expires_at                  │    │                   'kakao')  │
│ created_at                  │    │ provider_user_id            │
│ revoked_at      NULLABLE    │    │ created_at                  │
└────────────────────────────┘    └────────────────────────────┘

users → expenses (다른 모듈)  ← 인증된 user_id 만 받음
```

## 보안 정책

| 항목 | 정책 |
|------|------|
| 비밀번호 | bcrypt 해싱 (cost 12), 평문 저장 금지 |
| 세션 토큰 | JWT 또는 secure httpOnly 쿠키 |
| 통신 | HTTPS 필수 (배포 환경) |
| 입력 검증 | Pydantic (서버) + zod (클라이언트) |
| 무차별 대입 방어 | 로그인 5회 실패 시 5분 잠금 |
| 타이밍 공격 방어 | 로그인 실패 시에도 bcrypt verify 실행 (이메일 존재 유출 방지) |
| CORS | 운영 도메인 명시 (`*` 금지) |
| 회원 탈퇴 | Soft delete 30일 보관 → 자동 hard delete (실수 방지) |
| 개인정보 | 이메일 외 PII 최소화. 결제 정보·전화번호 미수집 |

## 회의에서 결정 필요 (5개)

1. **인증 방식**:
   - (a) 이메일+비밀번호 만 → 가장 단순, 비밀번호 재설정 흐름 필요
   - (b) Google OAuth 만 → 가장 빠름, 시연 20명에 과해질 수 있음
   - (c) 둘 다 + 카카오 → 사용자 선택권, 구현 비용 ↑
   - **추천**: 시연 20명이면 (b) Google OAuth 만 (NextAuth 라이브러리 30분 셋업)

2. **회원 탈퇴 방식**:
   - Soft delete (30일 보관 후 hard) — 실수 복구 가능
   - Hard delete (즉시 삭제) — 단순
   - **추천**: Soft delete (졸작 어필 + 안전)

3. **프로필 항목**:
   - 닉네임만 → MVP
   - 닉네임 + 아바타 → 게임화 어필
   - **추천**: 닉네임 + 아바타 (이미지 업로드 졸작 차별화)

4. **친구 기능**:
   - 졸작 범위 **밖** (Phase 2 — 시연 후)
   - 졸작 범위 **안** (졸작 어필 ↑, 일정 ↑)
   - **추천**: 범위 밖 (회원 모델은 친구 확장 가능하게 설계만 해두기)

5. **레벨·스탯 시각화**:
   - 회원 기능 안에 포함 (가입 시 Lv.1 발급) — 추천
   - 별도 게임화 모듈 분리

## 본 모듈이 다른 모듈에 제공하는 것

```
[회원 기능 모듈]
        │
        │ 인증된 user_id 발급
        ▼
[지출 입력 모듈] / [판정 모듈] / [히스토리·통계 모듈]

```
